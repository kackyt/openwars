# マスターデータアーキテクチャ

## 課題
現在、ゲームは `engine/src/resources/mod.rs` 内にハードコードされたデータプロパティ（例: `Terrain::defense_stars()`）に依存しています。しかし、ゲームデータは `engine/src/resources/master_data/*.csv` （およびマップデータは `map/` 階層下）に配置されたCSVファイルで詳細に定義されています。これらのCSVファイルをパースし、ECSのシステムやドメインロジックからアクセス可能にする必要があります。

## 解決策
1. **データモデル**: 各CSVファイルのレコードに対応する、Serde互換の構造体（例: `UnitRecord`、`TerrainRecord`）を定義します。マップのグリッドデータに対しては2次元配列としてパースする専用のパーサーを定義します。
2. **マップデータのデコード**:
   マップデータの各セルは数値で表現されていますが、この数値は「所有プレイヤー」と「地形ID」の両方を含んでいます。
   - 地形ID = `Cell Value % 100` （`landscape.csv`のIDに対応）
   - 所属プレイヤーID = `Cell Value / 100` （0の場合は中立、1はプレイヤー1、2はプレイヤー2...）
   これにより、例えば `101` は「プレイヤー1の首都(`1`)」、`203` は「プレイヤー2の工場(`3`)」、`12` は「中立の海(`12`)」としてパース時にデコードして保持します。
3. **パーサー**: エンジン起動時に、`csv` クレートと `include_str!` または `std::fs::File` を使用してレコードをデシリアライズします。
4. **リポジトリ/リソース**: パースしたデータを `MasterDataRegistry` 構造体に格納し、これをBevyの `Resource` として追加します。マップデータなど複数定義される要素は、ファイル名やIDをキーとしたコレクションとして保持させます。
5. **DDDとの統合**: コンポーネントやユースケースは、現在の列挙型ベースのハードコードされたロジックに代わり、移動コスト、防御ボーナス、武器のダメージなどのパラメータを `MasterDataRegistry` に照会します。

## 代替案の検討
- **JSON/RON**: 本プロジェクトにはすでに目的に適した適切に構造化されたCSVファイルが存在するため、CSVを使用することが明確な方針となります。そのためJSON/RONへの移行は却下します。
