# OpenWars AIアーキテクチャ・基本設計書

## 1. 目的とビジョン (Vision & Mission)
本ドキュメントは、ターン制戦略シミュレーションゲーム `openwars` において、「人間プレイヤーを圧倒しうる強く、かつ理不尽ではない思考ルーチン」を構築するための永続的なアーキテクチャ設計と長期的なロードマップを定義する。
当面はキャラクター（ユニット）の行動決定に集中するが、最終的には生産（拠点運用）を含めた大局的な戦略AIの完成を見据える。

## 2. 大会AI (TUBSTAP GAT2018) の分析からの学び
GAT2018等で活躍した強力なAIは、以下の要素を巧みに組み合わせることで、ターン制戦略特有の「組み合わせ爆発（Branching Factorのインフレーション）」を克服していた。我々もこのアプローチを継承・進化させる。

### A. 細分化されたモンテカルロ木探索 (Modified UCT)
1ターンの全ユニットの行動を1ノードとすると木が広がりすぎるため、「どのユニットを動かすか」「どう動かすか」という **個々のユニットアクション単位でノードを分割** して木を構築（M-UCTのアプローチ）。
### B. ドメイン知識による強烈な枝刈り (Heuristic Pruning)
「自分が受ける被害想定額（被ダメ×自価値）が敵に与える被害想定額（与ダメ×敵価値）を上回る攻撃」は無謀として探索ツリーへの追加すら行わない。ランダムな探索空間を「意味のある手」だけに絞ることでMCTSの実効性を高めている。
### C. ヘビー・プレイアウト (Heavy Playout)
MCTSの終端からのランダムシミュレーションにおいて、純粋なランダムではなく「20%の確率で攻撃可能な敵を殴る」等の軽量ヒューリスティクスを混ぜることで、探索が明後日の方向へ進むのを防ぎ、収束を早めている。

## 3. OpenWars AI アーキテクチャの長期設計

Rustの実行速度、およびECS（bevy_ecs）の並列処理能力を最大限に引き出し、TUBSTAPのAIを凌駕する思考数を確保するための設計とする。

### 3.1. AIステートの軽量化 (Lightweight Simulation State)
MCTSで1ターンに万単位のシミュレーション（プレイアウト）を回すため、GUIや描画ロジックを含む完全なECS Worldを毎回クローンするのはオーバーヘッドが大きい。
- **目標**: 探索ループ専用の軽量な状態構造体（例: `AiSimulationState`）を定義し、ユニット配置、HP、拠点状態など必要最小限のデータのみでシミュレーションを実行可能にする。

### 3.2. 静的盤面評価関数 (Static Evaluation)
MCTSのプレイアウトの終端や、貪欲法の基準となる高精度な盤面評価関数を構築する。
- **戦力評価**: `Σ (自軍ユニット現在HP × 動的価値) - Σ (敵軍ユニット現在HP × 動的価値)`
  - **動的価値 (Dynamic Value)** とは、単なる生産コストではなく状況に応じた補正を掛けた値とする。
    - **基準値**: そのユニットの生産コスト（マスターデータのコスト係数など）。
    - **相性補正**: 盤面に残存している敵ユニットの兵種構成に基づく補正。相手に「対空ミサイル」が多い場合は自軍の「戦闘ヘリ」の価値を大きく割り引き、逆に相手から弱点をつかれない兵種は価値を上げるなど。
    - **機動力・位置補正**: 足の遅いユニット（戦艦など）が主戦場から遠すぎる場合や、弾薬・燃料が枯渇している場合は「実質的戦力外」として価値を下げる。
    - **任務補正**: 拠点占領中の歩兵などは戦術的価値が極めて高いためボーナスを与える。
- **拠点プレッシャー**: 対象拠点の制圧進行度合い、拠点までのマンハッタン距離の逆数。
- **ダメージ予測**: ZOC（Zone of Control）や射程を考慮し、次ターンで確実に落とされるユニットへのペナルティ。

### 3.3. 並列・拡張可能なAIモジュール設計
- `engine/src/ai` モジュール下に、異なる思考アルゴリズムを切り替え可能な `AiAgent` トレイトを定義。
- `rayon` などの並列処理ライブラリを前提とし、複数のツリー探索やシミュレーションをマルチスレッドで完結させる設計とする。

### 3.4. 実験・検証用途の MCP サーバー (MCP Server for Validation)
- M-UCTの巨大なツリー構造や、動的価値の計算式調整結果を CLI 上のみで検証することは極めて困難である。
- そのため、GUI専用ビューアを作る代わりに **MCP (Model Context Protocol) サーバー** を構築する。これにより、LLMエージェントが直接ゲームの内部状態（評価値配列や合法手リスト）をJSONとして取得し、チャット上で可視化・対話的デバッグを行えるモダンなAI開発環境を実現する。

## 4. ロードマップと優先タスク（長期的視点）

AIの構築は複雑度が高いため、以下のフェーズに分けて段階的に実装と強化を行う。

### Phase 1: ベースラインAIの構築（ヒューリスティクスと貪欲法）
**目的**: AIが「ランダムではなく意味のある手」を打つ土台を作る。
- 静的盤面評価関数と「無謀な行動の枝刈りフィルター」を実装。
- すべての未行動ユニットが、評価関数によって自律的に最善手（局所解）を選ぶアルゴリズムを構築し、CLIとのターン制御連携を完了させる。
- **Phase 1.5: 戦略レイヤーの導入（マクロ目標とミッション管理）**
  - 貪欲法（局所解）では原理的に解決不可能な「離島への部隊輸送」など、複数ターンにまたがる協調行動を実現するため、階層型AIアーキテクチャ（戦略レイヤーと戦術レイヤー）を導入する。
  - 戦略レイヤーは以下の3つのサブフェーズに分けて段階的に実装する：
    - **Step 1: 島の認識と輸送ミッション基盤 (最小実装)**
      - フラッドフィルによる島（地形のまとまり）の分析と、`Pickup` → `Transit` → `Drop` → `Return` のフェーズを持つ輸送ミッション（TransportMission）の導入。貪欲法より優先してミッション行動を評価する。
    - **Step 2: 戦略プランナーの強化 (目標管理と連携)**
      - 拠点制圧などの「目標（Objective）」を管理し、島の規模や目標の優先度に応じた複数ユニットの割り当て、待機・占領行動の連携を行う。
    - **Step 3: 生産統合とミッションリカバリ**
      - 輸送機や歩兵が撃墜された際のミッション再割り当てや、戦略目標に基づく動的なユニット生産指示（Production）を統合する。
*(※ 現在進行中のタスク領域)*

### Phase 2: 戦術的探索の導入（ビーム探索 / ユニット連携の基本）
**目的**: ユニット単体ではなく、部隊としての連携（押し出し、退避、合流）を可能にする。
- フェーズ1を拡張し、ユニットの行動順序の並び替えパターン（順列）を考慮する。
- 1ターン全体の推移において、上位Nパターンの状態を保持し続けるビーム探索（Beam Search）エンジンを実装する。これにより、狭い通路でのユニットの入れ替わり攻撃等を実現する。

### Phase 3: 大会AIレベルへの到達（超並列M-UCTの実装）
**目的**: 先を見据えた高度な戦略行動を獲得し、人間を超える戦術計算能力を実現する。
- TUBSTAPの強力なAIアルゴリズムである **M-UCT（ユニット行動分離型モンテカルロ木探索）** をベースにしたAI Agentを実装。
- Phase 1で育成した静的評価関数をプレイアウトの終端評価として用い、Phase 1の貪欲行動をプレイアウト中の低確率行動として採用する（強力なヘビー・プレイアウト）。
- Rustの並列計算タスクとして数万回レベルのノード探索を数秒で完了させる並列木探索システムを完成させる。

### Phase 4: 生産拡張とマクロ戦略（全体最適AI）
**目的**: ユニットの移動・攻撃（マイクロ）だけでなく、生産・補給・資金管理（マクロ）を統合する。
- 生産フェーズの木展開。現在の手持ち資金と前線の状況から、「防衛のための安い歩兵」か「突破のための戦車」かをMCTSのノードに含める。
- 「敵の首都の占領」と「敵部隊の殲滅」という勝利条件に対する長期的プランニング能力を搭載する。
